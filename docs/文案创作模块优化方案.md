## 一、问题背景

### 1.1 当前文案创作页的输入结构

现有页面（`/create/text/page.tsx`）的输入字段：

| 字段 | 必填 | 说明 |
|------|------|------|
| 所在行业 | ✅ | 下拉选择：餐饮/酒店/休闲娱乐 |
| 品类 | ✅ | 下拉选择：火锅/烤肉/奶茶等 |
| 营销主题/创作需求 | ✅ | Textarea，500字以内自由描述 |

**核心问题**：品类信息过于粗糙（"奶茶"是奶茶，"廿四黑陈皮柠檬茶"也是奶茶），且缺少门店身份信息，AI 无法区分"区域性老店"和"连锁新品牌"，生成的脚本千人一面、缺乏个性。

### 1.2 用户反馈定位

用户输入"奶茶 + 廿四黑陈皮柠檬茶" → 期待 → 围绕这杯具体产品的专属脚本  
实际输出 → 奶茶店通用脚本（开场："奶茶控们看过来！"）

**根因**：prompt 上下文不足，AI 只能靠品类泛化，导致生成内容与用户实际需求脱节。

---

## 二、优化目标

| 目标 | 描述 |
|------|------|
| 内容相关性提升 | 生成脚本与用户输入的匹配度从"品类级"提升到"门店/产品级" |
| 模块串联打通 | 文案生成 → 图片/视频创作的流转无缝，不割裂 |
| 降级兜底保障 | 搜不到门店信息时，有品类优秀案例作为 fallback，不空转 |

## 四、输入字段详细设计

### 4.1 字段清单与约束

| # | 字段名 | 类型 | 必填 | 最大长度 | 说明 |
|---|--------|------|------|----------|------|
| 1 | 文案类型 | 单选 | ✅ | — | 短视频脚本 / 直播话术 / 种草文案 / 评论回复 |
| 2 | 门店名称 | 文本输入 | ✅ | 50字 | 输入后触发联网搜索；搜索建议下拉 |
| 3 | 所在行业 | 下拉 | ✅ | — | 餐饮美食 / 酒店民宿 / 休闲娱乐 |
| 4 | 品类 | 联动下拉 | ✅ | — | 随行业联动（二级联动） |
| 5 | 营销目标 | 单选 | ✅ | — | 拉新 / 复购 / 促销 / 品宣 / 自定义 |
| 6 | 品牌简介/产品特色 | 多行文本 | 选填 | 800字 | Textarea；搜索成功后可预填充 |
| 7 | 搜索来源信息 | 只读展示 | — | — | 搜索成功后在 UI 展示，不可编辑 |

### 4.2 字段交互规则

**门店名称输入框（核心改动）**

```
用户输入 → 停顿 300ms（防抖）→ 调用搜索接口 → 下拉展示搜索建议
                                              │
                                    ┌─────────┴─────────┐
                              选择建议门店          输入新名称（无匹配）
                                    │                     │
                                    ▼                     ▼
                            展开门店信息卡片        显示"未找到门店"提示
                            (评分/招牌/评价摘要)     引导用户补充品牌简介
                                    │                     │
                                    ▼                     ▼
                            用户可修改预填充内容      用户补充品牌简介
```

**品牌简介字段的填充逻辑**

| 场景 | 预填充内容来源 | 用户可修改 |
|------|---------------|-----------|
| 搜索到门店 | 搜索接口返回的招牌产品、评价摘要、品牌描述 | ✅ 可手动修改 |
| 未搜到门店 | 空（placeholder 引导填写） | ✅ 纯手动输入 |
| 用户跳过搜索 | 使用品类通用默认值 | ✅ 纯手动输入 |

**营销目标字段说明**

| 目标 | 对应 prompt 策略 | 示例内容方向 |
|------|-----------------|-------------|
| 拉新 | 强调"第一次来"的惊喜感、分享动机 | "还没来过的宝子们..."、"XXX一次就爱上" |
| 复购 | 强调老客福利、会员权益、情感连接 | "老朋友们，这个月的新品..."、"会员专属..." |
| 促销 | 突出价格力、限时紧迫感 | "仅限X天"、"这个价格错过等一年" |
| 品宣 | 品牌故事、差异化、价值观 | 创始人故事、食材溯源、匠心工艺 |
| 自定义 | 开放 prompt，让用户自由描述 | 跟随用户输入的描述方向 |

## 五、联网搜索增强方案

### 5.1 搜索接口设计

**搜索触发时机**：用户输入门店名称后，停顿 300ms 自动触发（debounce），或用户手动点击搜索按钮。

**并行搜索策略**：同时向抖音、小红书、大众点评三个平台发起搜索，取置信度最高的结果。

### 5.2 Prompt 注入策略

**Prompt 构建顺序**（由高到低覆盖）：

1. 门店真实信息（来自搜索接口）
2. 用户补充的品牌简介
3. 品类通用知识（模型自身能力）
4. 营销目标约束
5. 文案类型约束

**Prompt 示例（短视频脚本，搜索到门店）**：

```
你是一名抖音生活服务内容专家。请为以下门店生成短视频脚本。

【门店信息】
- 门店名称：廿四黑·陈皮柠檬茶
- 行业：餐饮-茶饮
- 评分：4.7（大众点评，1234条评价）
- 招牌产品：陈皮柠檬茶、茉莉陈皮饮
- 品牌特色：0蔗糖配方，陈皮产自新会，小红书爆款
- 用户真实评价："陈皮柠檬茶超好喝，茶味浓郁，夏天解腻神器"

【营销目标】
拉新——吸引第一次听说这个品牌的用户到店

【文案类型】
短视频脚本（15-60秒），剧情型，强转化

【输出要求】
- 开场：3秒内抓住注意力，抛出一个痛点或悬念
- 主体：围绕招牌产品展开，有具体场景和动作描述
- 结尾：明确的行动号召（到店/下单/评论）
- 包含字幕提示和配音台词
```

    setUseTemplate(false); // 切换到手动模式
  }

## 七、数据结构设计

## 八、推荐接入的搜索/数据源 API

### 8.1 搜索服务优先级

| 优先级 | 数据源 | 推荐理由 | 预估成本 |
|--------|--------|----------|----------|
| P0 | 大众点评开放平台 | 门店基础信息最全（评分/评价/招牌菜），API 合规性好 | 中（有免费配额） |
| P0 | 抖音开放平台（商家号） | 商家官方数据，可直接获取门店信息 | 低（有商家授权情况下） |
| P1 | 小红书商业数据 API | 种草氛围感强，利于提取用户真实评价和卖点 | 高（需商业授权） |
| P2 | 微信公众号搜一搜 / 腾讯地图 | 补充数据源，覆盖更多本地商家 | 低 |
| P3 | 自建搜索索引 | 将历史用户输入的优质门店信息沉淀为搜索候选 | 一次性建设成本 |

### 8.2 各平台 API 对接概览

#### 大众点评开放平台（推荐优先接入）

- **API 名称**：门店搜索 / 门店详情
- **申请地址**：https://open.dianping.com/
- **数据覆盖**：评分、评价数、人均价格、招牌菜、用户评价摘要
- **配额**：个人开发者有免费基础配额，商业应用需申请企业认证
- **接入难度**：⭐⭐（有标准 REST API，文档完善）

#### 抖音开放平台

- **API 名称**：门店信息查询 / POI 详情
- **申请地址**：https://open.douyin.com/
- **数据覆盖**：门店基本信息、商家认证状态、用户互动数据
- **配额**：需申请 relevant scope 权限
- **接入难度**：⭐⭐⭐（需商家完成抖音认证授权）

#### 小红书（无官方公开 API）

- **方案 A**：红薯智云（第三方数据服务，有数据接口）
- **方案 B**：自建爬虫（⚠️ 违反平台 TOS，不推荐）
- **方案 C**：让用户主动授权小红书账号（参考 LinkOpen 模式）
- **接入难度**：⭐⭐⭐⭐（无官方 API，需绕道）

## 十一、风险与注意事项

| 风险项 | 影响 | 应对措施 |
|--------|------|---------|
| 搜索 API 延迟导致用户体验差 | 用户输入后等待时间过长 | 加 loading 骨架屏 + 5s 超时自动降级到案例库 |
| 搜索结果不准确（同名门店） | 生成内容与用户实际门店不符 | 展示置信度分值 + 用户可修改预填充内容 |
| 第三方 API 成本不可控 | 大规模用户使用后 API 成本激增 | 设置每日搜索配额；关键信息走缓存 |
| Prompt 注入过长超出模型上下文 | 搜索结果太多导致 prompt 超限 | prompt 总长限制在 2000 tokens 以内，截断低优先级内容 |
| 门店信息隐私合规 | 展示门店地址/评价等可能涉及隐私 | 仅展示公开可见信息；加"信息来源：XXX平台"标注 |
